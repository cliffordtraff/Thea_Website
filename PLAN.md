# PLAN.md — Thea Traff Portfolio (Proof of Concept)

A clean, original rebuild of the editorial photography portfolio at `theatraff.com`.
This is an **experimental proof of concept**: it demonstrates that the site's
structure, editorial feeling, responsive behavior, and image-first presentation
can be recreated with a modern, maintainable stack — **without copying any of the
reference site's generated code, markup, class names, or photographs.**

The reference site was inspected only as a visual/structural reference (via
headless Playwright). All photographs are replaced with clearly labeled neutral
placeholder blocks.

---

## 1. Architecture

- **Framework:** Next.js (App Router) + TypeScript.
- **Rendering:** Server Components by default. No client-side JavaScript is
  required for the current feature set (navigation is plain links, layouts are
  static). Client Components are introduced only if an interaction demands it.
- **Styling:** A single global stylesheet of design tokens
  (`app/globals.css`) plus **CSS Modules** co-located with each component.
  No Tailwind, no CSS-in-JS runtime.
- **Images:** `next/image` with explicit `width`/`height` (true aspect ratios).
  Placeholder assets are flat-tone SVGs generated by a script.
- **Fonts:** `next/font/local` or `next/font/google` (self-hosted, no external
  requests, no layout shift). See `DECISIONS.md` for the specific face.
- **Deployment target:** Vercel-compatible (standard Next.js output).

### Dependency policy
`next`, `react`, `react-dom`, `typescript`, and type packages only. Every new
dependency must be justified in `DECISIONS.md`. (Playwright is a **dev-only**
tool for visual verification and lives outside the app bundle.)

## 2. Page structure (routes)

| Route              | Page                | Reference source        | Layout family            |
| ------------------ | ------------------- | ----------------------- | ------------------------ |
| `/`                | Home                | `Home-Page`             | Overlaid grid index      |
| `/commissions`     | Recent Commissions  | `Recent-Commissions-1`  | Sidebar + image sequence |
| `/personal`        | Personal Work       | `Overview`              | Sidebar + image sequence |
| `/elevator-series` | Elevator Series     | `Elevator-Series`       | Sidebar + image sequence |
| `/info`            | Info + Contact      | `Info-Contact`          | Sidebar + portrait + bio |

Two shared page **shells**:

1. **Overlay shell** (Home) — full-bleed grid of images; the site title and
   navigation are overlaid on the first cell. A quiet copyright footer.
2. **Sidebar shell** (all interior pages) — a fixed left navigation column
   (~250px) with the title + nav; the content column to the right holds an
   editorial sequence of image-layout blocks. The active nav item is dimmed.

Both shells collapse gracefully on mobile (nav becomes a stacked top header;
images reflow to fewer columns / single column).

## 3. Component structure

```
components/
  Nav.tsx                 # server; renders title + links, takes `active` prop
  SiteFrame.tsx           # server; sidebar shell wrapper for interior pages
  HomeGrid.tsx            # server; the overlaid home grid
  Figure.tsx              # server; base image primitive (next/image + <figure>)
  Caption.tsx             # server; restrained "^ subject / PUBLICATION" caption
  ProjectSequence.tsx     # server; maps a content block list -> layout primitives
  layouts/                # editorial layout variants (see §5)
    FullWidth.tsx  WideLandscape.tsx  MediumLandscape.tsx
    LargePortrait.tsx  SmallPortrait.tsx  CenteredPortrait.tsx
    OffsetLeft.tsx  OffsetRight.tsx
    ImagePair.tsx  StackedPair.tsx  Triptych.tsx
```

Every primitive is a thin, composable Server Component. Pages compose them from
data — no universal gallery component.

---

## 4. Content model

Portfolio content is **fully separated from rendering**. Typed data lives in
`content/`; components only know how to render a typed block.

```
content/
  types.ts        # ImageAsset, Caption, LayoutBlock (discriminated union), Project, Page
  site.ts         # site metadata + nav items
  home.ts         # ordered image cells for the home grid
  commissions.ts  # Recent Commissions project(s) -> ordered layout blocks
  personal.ts     # Personal Work -> ordered layout blocks
  elevator.ts     # Elevator Series -> ordered layout blocks
  info.ts         # bio, contact (placeholder), clients, portrait
```

Key types (see `content/types.ts` for the source of truth):

- `ImageAsset` — `src`, `width`, `height`, `alt`, optional `placeholder` flag.
- `Caption` — `subject`, optional `publication`/`credit`, optional `year`.
- `LayoutBlock` — a **discriminated union** keyed by `variant` (the layout
  vocabulary in §5), each carrying the image(s) + optional caption it needs.
- `Project` — `title`, `slug`, ordered `blocks: LayoutBlock[]`.

Adding or resequencing a project means editing data, never components.

## 5. Editorial layout vocabulary

A small, deliberate set of primitives — **not** one universal grid:

| Variant             | Intent                                             |
| ------------------- | -------------------------------------------------- |
| `full-width`        | Edge-to-edge within the content column             |
| `wide-landscape`    | Large landscape, dominant                          |
| `medium-landscape`  | Mid-scale landscape                                |
| `large-portrait`    | Tall dominant portrait                             |
| `small-portrait`    | Small portrait, lots of surrounding space          |
| `centered-portrait` | Single portrait centered in the column             |
| `offset-left`       | Image pushed toward the left margin                 |
| `offset-right`      | Image pushed toward the right margin                |
| `image-pair`        | Two images side by side (asymmetric widths allowed)|
| `stacked-pair`      | Two images stacked, offset from each other          |
| `triptych`          | Three images in a row                              |

Scales and offsets are driven by CSS custom properties (the width categories in
`DESIGN_SYSTEM.md`) so compositions stay consistent yet varied.

## 6. Implementation phases

1. **Inspect & plan** — inspect reference (done via Playwright), write these docs.
2. **Foundation** — scaffold, routes, tokens, both nav shells, empty page shells,
   confirm production build.
3. **Visual POC** — shared shell + representative home + a full commission section
   + ≥4 distinct layout treatments, desktop + mobile, placeholder captions/metadata.
4. **Verify** — Playwright screenshots at 1440×1000, 1024×900, 768×1024, 390×844;
   console/network checks; overflow + aspect-ratio checks; `next build` + lint.

---

## 7. Assumptions

- **No supplied assets.** The project folder was empty; all imagery is neutral
  labeled placeholder blocks at realistic aspect ratios. Real photographs are
  never scraped or republished.
- **Contact details are placeholders.** The reference Info page exposes a real
  email/phone; those are private and are replaced with clearly marked placeholders.
- **Font is a stand-in** for the reference's licensed *Sabon*. See `DECISIONS.md`.
- **Background color and measurements are approximated** from screenshots, not
  copied from the reference's stylesheet.
- Reversible design guesses are logged in `DECISIONS.md` rather than made silently.
